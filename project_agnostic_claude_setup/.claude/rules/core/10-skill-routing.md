# Process Before Execution

Use process surfaces before coding when the task is not obviously small.

- unclear request or open-ended feature: brainstorm first
- approved requirements plus multi-step code work: plan first
- newly written requirements or plan docs: review the document before execution
- risky behavior change or fragile legacy area: use test-driven-development or characterization-first execution
- bug, failure, or confusing behavior: use systematic-debugging before proposing fixes
- approved plan with bounded tasks: use subagent-driven development
- about to claim success: verify before completion

Process guidance comes before implementation guidance.

If the user already supplied a clear, approved plan, do not re-run brainstorming.
